# D-ONWORKS 회사 공통 규칙

> 이 파일은 모든 D-ONWORKS 프로젝트의 `.claude/rules/` 에 symlink로 연결됩니다.
> 수정 시 전 프로젝트에 즉시 반영되므로 신중하게 변경하세요.

## 언어
- 모든 설명, 주석, 커밋 메시지는 **한글** 작성. 변수명·함수명·파일명만 영어.
- 코드 주석도 한글로 충분히: 클래스 역할, public 메서드 설명, 복잡한 로직 단계별 설명.

## AI 코드 품질 (바이브 코딩 핵심)
- AI가 추천한 패키지는 반드시 `npm info`/`pip show`로 **실존 여부 확인** (환각 패키지 19.7% 발생)
- 같은 기능에 서로 다른 라이브러리 혼용 금지 (axios+fetch, moment+dayjs 등)
- 미사용 import/변수/함수, 디버깅 잔재(`console.log`), 주석 처리된 코드 블록 즉시 제거
- AI 코드를 검증 없이 수락 금지. **"잘 작동합니다" 금지** — 수치 증거(건수, 성공/실패) 필수
- 같은 파일 fix 2회 이상 → 근본 원인 분석 먼저. 임시 fix 쌓지 말 것

## 코드 구조
- 모든 파일 **500줄 초과 금지**. 초과 시 가급적 분리
- 설정값/상수는 코드에 직접 넣지 말고 환경변수·설정파일로 외부화 (하드코딩 금지)
- 에러를 삼키는 코드 금지: `catch (e) { return 기본값 }` 패턴 사용 금지. 에러는 상위로 전파하거나 명시적으로 로깅할 것

## 보안 위생
- `.env` 파일은 반드시 `.gitignore`에 포함. 환경변수 누락 시 앱 시작 실패(fail-fast) 처리
- 모든 외부 입력(body, query, params)은 스키마 검증 필수. SQL은 파라미터 바인딩만 허용
- 에러 응답에 스택 트레이스, DB 정보, 내부 에러 메시지 노출 금지
- `req.body`를 직접 DB에 전달 금지 (Mass Assignment 방지)

## 작업 프로세스
- 구현 후 사용자에게 **구체적 확인 포인트** 안내 ("이 버튼 클릭해서 확인해보세요")
- 에러 발생 시 해결책을 함께 제시: "에러가 났는데, 이렇게 고치면 됩니다" 형태
